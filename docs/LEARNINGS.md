# LEARNINGS — грабли и уроки реального развёртывания

Записано по итогам первой боевой установки (сервер EDIS Латвия + MacBook Apple
Silicon). Чтобы будущие сессии знали контекст и не наступали на те же грабли.

## 1. `curl | grep -m1` под `set -o pipefail` валит скрипт
Симптом: `curl: (23) Failure writing output to destination, passed N returned 0`.
Причина: `grep -m1` закрывает пайп после первого совпадения → curl получает broken
pipe → под `pipefail` весь пайплайн возвращает ошибку → `set -e` рвёт скрипт.
Фикс: качать в файл, потом грепать файл (`curl -o f.json ...; grep ... f.json`).
Исправлено в `scripts/setup-singbox-latvia.sh` (`install_singbox`).

## 2. Длинные многострочные вставки в Терминал ломаются
Bracketed paste (`^[[200~`), перенос строк и heredoc рвут вставку больших блоков.
Фикс: использовать `git clone` + запуск скрипта, либо однострочные команды.

## 3. Почта/markdown превращает голые домены в ссылки
`www.microsoft.com` → `[www.microsoft.com](https://www.microsoft.com)` при копировании
через письмо/markdown-рендер → ломает команды и значения.
Фикс: передавать через Telegram «Избранное» (plain text) или печатать руками;
значения вроде SNI задавать явно и проверять.

## 4. VMware Horizon: буфер обмена односторонний
local→remote разрешён, remote→local запрещён политикой компании (DLP).
Обход: переслать текст себе через сервис, открывающийся и локально (почта/Telegram).

## 5. Контекст пользователя: читает инструкции В Horizon, выполняет ЛОКАЛЬНО
Claude локально заблокирован → инструкции читаются в удалённом столе, а команды
выполняются в локальном терминале Mac. Отсюда трудности переноса команд (см. #3, #4).

## 6. EDIS: на сервере нет `ufw`
Скрипт это переживает (ветка else). Порты 443 tcp/udp открыты у провайдера по
умолчанию — отдельно открывать не пришлось.

## 7. sing-box 1.13: обязателен `route.default_domain_resolver`
`sing-box check` падает FATAL: "missing route.default_domain_resolver ... 1.12.0".
Фикс: добавить в `route` клиентского конфига `"default_domain_resolver": "dns-local"`.
NB: именно `dns-local` (а не через прокси) — чтобы не было кольца: urltest резолвит
тестовый URL, а прокси в этот момент ещё тестируется.
Исправлено в `configs/singbox-client.template.json`.

## 8. REALITY_SNI вставился со скобками markdown (следствие #3)
В конфиг попало `[www.microsoft.com](https://...)`. Клиентский SNI ОБЯЗАН совпадать
с серверным (`www.microsoft.com`). Фикс на месте — perl-замена в local-конфиге.

## 9. rule_set `geoip-telegram` → 404 → краш-луп sing-box
Симптом: `FATAL start service: initialize rule-set[1]: geoip-telegram: unexpected
status: 404 Not Found`. У SagerNet/sing-geoip нет `geoip-telegram.srs` → 404 →
sing-box не стартует → TUN не создаётся → трафик идёт напрямую (страна = RU).
Фикс: убрать rule_set `geoip-telegram` и его route-правило. Telegram остаётся по
`domain_suffix` (telegram.org, t.me, ...). В full-режиме app-трафик Telegram идёт
через `final: proxy`.
Исправлено в `configs/singbox-client.template.json`.

## 10. Авто-режим по сети — УДАЛЁН (был footgun)
Авто-режим (LaunchAgent, переключал full/selective по сети) при дефолте
`selective` сам возвращал режим в selective на незнакомых сетях → обычные сайты
показывали RU, пользователь воспринял как «утечку». Для цели «всегда через Латвию»
эта фича только мешала. Решение: **полностью удалена** из проекта (файлы
`vpn-autonet.sh`, `install-autonet.sh`, `com.user.singbox-autonet.plist`,
`netmap.example`, ссылки в меню/`vpn.sh`/инсталляторах). Вывод на будущее: не
включать «умные» авто-переключатели режима по умолчанию — для VPN предсказуемость
важнее «умности».

## 11. Telegram-приложение ходит по IP — нужен ip_cidr, не rule_set
Вместо сломанного `geoip-telegram` (см. #9) добавлено правило `ip_cidr` с
официальными диапазонами Telegram (`91.108.x`, `95.161.64.0/20`, `149.154.160.0/20`,
IPv6 `2001:67c:4e8::/48` и т.д.) → `outbound: proxy`, ПЕРЕД `geoip-ru` (чтобы
Telegram шёл через туннель даже если IP попадает в RU-диапазон). Надёжнее внешнего
rule_set (не зависит от доступности/404).

## Важно про kill-switch и режимы
Kill-switch (pf) рассчитан на FULL: блокирует весь egress кроме utun + server:443 +
LAN. В режиме selective прямой (direct) трафик к произвольным зарубежным IP тоже
попадает под блок → selective + kill-switch могут конфликтовать. Вывод: для
«никогда не светить RU» правильный режим — **FULL + kill-switch ON** (зарубежное
всегда через туннель маршрутизацией; kill-switch ловит только падение туннеля).

## 12. Heredoc в пайплайне цепляется к ПОСЛЕДНЕЙ команде, не к первой
Симптом: установщик вместо генерации `config-strict.json` уронил пользователя в
интерактивный Python REPL, а в файл записался исходник питона.
Причина: `python3 - "$CFG" | sudo tee out.json <<'PY' ... PY` — heredoc `<<'PY'`
привязался к `sudo tee` (последний элемент пайпа), поэтому `python3` остался без
stdin → интерактивный режим, а `tee` записал текст heredoc как «данные».
Фикс: вешать heredoc на нужную команду — `python3 - "$CFG" <<'PY' | sudo tee out.json`.
Исправлено в `scripts/install-macos-daemon.sh` (генерация config-strict).

## 13. Переключение режима/протокола из меню падало на sudo → залипал спиннер
Симптом: клик по смене режима в тулбаре → меню «схлопывалось» в `⏳ Применяю…` и
висело так навсегда.
Две причины:
  (а) SwiftBar запускает действия НЕинтерактивно — sudo не может спросить пароль.
      В `/etc/sudoers.d/singbox` не было правил для нового strict-режима и записи
      маркера: добавить `cp config-strict.json -> config.json` и
      `tee /etc/sing-box/mode`. Без них `sudo` молча падал и переключение рвалось.
  (б) При `set -e` оборванный `apply()` пропускал `busy end` → `/tmp/vpn-busy` не
      удалялся → спиннер залипал.
Фикс: (а) добавить правила в `scripts/setup-sudoers.sh`; (б) `trap 'busy end' EXIT`
в `vpn-mode.sh`, `vpn-proto.sh`, `killswitch.sh` — спиннер снимается всегда.
Вывод: любая команда из меню, дёргающая sudo, ТРЕБУЕТ соответствующего NOPASSWD-
правила (точный путь+аргументы); проверять `sudo -n -l` после изменений.

## 14. Три режима маршрутизации (strict/full/selective) + персистентность
- `strict` — ВЕСЬ трафик через Латвию, включая RU (config-strict: из full убраны
  правила «RU→direct»: `geoip-ru` и `domain_suffix .ru/.рф`).
- `full` — умный: зарубеж через Латвию, RU напрямую (по умолчанию).
- `selective` — только сервисы через туннель (`final: direct`).
Режим хранится в `/etc/sing-box/mode` + активный `config.json` — оба на диске,
**переживают перезагрузку** (поднимается последний выбранный режим). НО повторный
запуск `install-macos-daemon.sh` сбрасывает на `full` (так задумано).

## 15. Проверка fail-closed kill-switch (как доказать отсутствие утечки)
`killswitch on` → `vpn off` → `curl --max-time 6 https://ipinfo.io/json`.
Правильный результат: `curl: (28) Connection timed out` (обрыв), а НЕ `country:RU`.
Значит при падении туннеля зарубежные сервисы получают «нет связи», а не реальный
IP. RU-сайты при этом продолжают работать напрямую (таблица `<ru>` в pf, ~8600
CIDR из ipdeny). Подтверждено вживую на боевой установке.

## Общий вывод по надёжности шаблонов
Удалённые rule_set'ы — потенциальная точка отказа (404/блокировка/недоступность на
старте). Если добавляем новый rule_set — проверять, что URL реально существует, и
помнить, что любой их сбой = FATAL у sing-box (краш-луп). По возможности —
минимизировать зависимость от внешних rule_set на старте.
