# CLAUDE.md — операционные заметки и грабли

Личный VPN: macOS/iOS клиент (sing-box) + VPS в Латвии (sing-box, VLESS+Reality на
443/tcp). Один протокол — Reality (Hysteria2 убран ради меньшей поверхности атаки).
Карта файлов — в `DEPLOYMENT.md`, инструкции — в `docs/`.

> Этот файл — память о реально потраченных часах отладки. Прочитай ПЕРЕД тем, как
> чинить «нет интернета с VPN», иначе наступишь на те же грабли.

## ⚠️ Грабля №1 — чат превращает домен в markdown-ссылку при копировании

Самая дорогая ловушка этого проекта. Когда значение вроде `www.microsoft.com`
проходит через чат/мессенджер, при копировании оно становится
`[www.microsoft.com](https://www.microsoft.com)`. Если это попадёт в конфиг
(`server_name`, SNI), Reality-хендшейк ломается, а `sing-box check` проходит
(строка валидна как JSON) — поэтому ошибку легко не заметить.

Правила:
- **Никогда не вводи и не вставляй домены руками** в конфиг. Читай их программно с
  сервера: `python3 -c '...print(...["tls"]["server_name"])'` и присваивай через
  переменную/argv, не печатая домен в команде.
- **Проверяй чисто ли значение** длиной/скобками, а не глазами (чат и вывод тоже
  испортит): `python3 -c '... s=...; print(len(s), "[" in s)'`.
  `www.microsoft.com` = **17 символов, без `[`**. Если `len` 44 и `bracket True` —
  значение испорчено markdown.
- В скриптах подстраховались regex-ом `\[([^\]]+)\]` для вырезания markdown.

## ⚠️ Грабля №2 — setup-singbox-latvia.sh раньше менял ключи при каждом запуске

`gen_secrets()` генерировал НОВУЮ Reality-пару/UUID/short_id на каждом прогоне →
повторный запуск на сервере молча ломал ВСЕ клиенты (в логе сервера:
`REALITY: processed invalid connection`). Теперь **идемпотентно**: переиспользует
существующие ключи и SNI из `/etc/sing-box/config.json` + `credentials.txt`. Свежие
ключи — только на первой установке. Всё равно: не дёргай setup без нужды.

## ⚠️ Грабля №3 — DDoS-защита провайдера банит ТВОЙ IP при флуде

EDIS (и подобные) при потоке подключений к серверу включают авто-митигацию и
**режут именно твой исходящий IP** по TCP/ICMP (UDP может проскакивать). Симптом:
сервер пингуется и коннектится по TCP **со всего мира, включая РФ** (check-host),
а у тебя с мака/телефона — `ping` 100% loss, `ssh`/443 timeout, но `traceroute`
доходит. Это **не РКН и не смерть сервера**.

Причина в этом проекте: отладка с включённым-но-сломанным туннелем → sing-box
долбил сервер сотнями переподключений/мин → EDIS принял за атаку.

Защита (внедрено): `vpn-watch.sh` теперь после 3 неудачных авто-рестартов
**выключает туннель** (circuit-breaker), чтобы не флудить. Не оставляй
включённым заведомо нерабочий туннель и не гоняй urltest вручную тысячи раз.

Восстановление доступа, если IP уже забанен:
1. **Ребут домашнего роутера** → новый IP (часто спасает сразу).
2. **Тикет в EDIS** с просьбой снять митигацию/вайтлистнуть IP.
3. **Подождать ~1–6 ч** — авто-бан сам спадает после прекращения «атаки».
   Доступ к серверу всё это время есть через **VNC/Open Console в панели EDIS**.

## Диагностика «нет интернета с VPN» — порядок, отсекающий причины

1. **Демон жив?** `pgrep -x sing-box`; лог `tail -n 30 /var/log/sing-box.err.log`.
   - `no available network interface` (iOS, на сотовой) — потеря интерфейса при
     смене сети; переподключить VPN. На IPv6-only сотах бывает из-за IPv4-сервера.
   - `outbound/...: EOF` — соединение с сервером рвут (либо сервер отвергает
     хендшейк, либо блокировка по пути).
2. **Сервер отвергает или путь?** Серверный лог:
   `ssh root@SRV 'journalctl -u sing-box -n 20 --no-pager'`.
   - `REALITY: processed invalid connection` от твоего IP = сервер НЕ авторизует
     клиента → рассинхрон параметров (ключи/SNI/short_id) ИЛИ битый SNI (грабля №1).
3. **Это блокировка по сети или конфиг/сервер?** Запусти Reality
   **на самом сервере по `127.0.0.1`** (см. ниже). Локальный тест идёт по петле —
   DPI/блокировки там физически нет. Если и локально `invalid connection` → дело в
   сервере (конфиг/бинарь), НЕ в блокировке.
4. **Это РКН-блок IP сервера или бан твоего IP?** `check-host.net`:
   - **TCP port** `SRV:443` (не только Ping/ICMP!). Если РФ-ноды Connected, а у тебя
     нет → банят **твой** IP (EDIS, грабля №3). Если РФ-ноды fail, а зарубеж OK →
     РКН режет **IP сервера** → нужен новый IP сервера.

### Локальный тест Reality на сервере (петля, без сети)
Поднять на сервере минимальный Reality сервер+клиент на `127.0.0.1` и прокачать
`curl --socks5-hostname`. Если `country` приходит — Reality на сервере рабочий,
проблема в сети/клиенте; если `EOF`/`invalid` — проблема на сервере. SNI и ключи
брать ИЗ конфига (не печатать домены руками). Рабочие сниппеты — в истории; ключи
сверять по факту парности, а не «по бумаге».

### Сверка параметров клиент↔сервер (через ssh, VPN выключен)
`uuid`, `short_id`, `server_name`(SNI), `flow`, `public_key`(клиент) vs
`private_key`(сервер, пара к публичному). Версии sing-box на обоих концах. Длины:
Reality pub/priv = 43 символа base64; SNI = 17 для www.microsoft.com.

## Открытый вопрос (не дорешён в той сессии)

Даже со свежей гарантированно-парной парой ключей, чистым SNI и достижимым
прикрытием минимальный Reality падал на `127.0.0.1` с `invalid connection`
(sing-box 1.13.13 с обеих сторон, flow `xtls-rprx-vision`). Не довели из-за бана IP.
**Следующие шаги, когда доступ вернётся:**
- доделать минимальный тест **без `flow`** (изолировать vision) — если без flow
  работает, убрать `xtls-rprx-vision` из конфигов сервера и клиента;
- если и без flow падает — **переустановить/откатить sing-box** на сервере;
- проще и надёжнее — **чистая пересборка**: EDIS «Reinstall OS» → Debian → один
  прогон `setup-singbox-latvia.sh` (теперь идемпотентный) → синхрон клиента.
  Делать аккуратно, БЕЗ повторного флуда (иначе снова бан IP).

## Полезные факты окружения
- VPS: EDIS Global, `192.36.41.201`, Латвия, 1 GB RAM (легко перегрузить — не
  гонять по нескольку sing-box разом). Панель: power/VNC/Reinstall, Support PIN.
- Конфиг сервера: `/etc/sing-box/config.json`; секреты: `/etc/sing-box/credentials.txt`.
- Mac: демон `com.user.singbox` (LaunchDaemon), управление — алиас `vpn`/меню SwiftBar.
- Горизонт (рабочий VPN) и наш туннель **несовместимы** — переключаться вручную;
  kill-switch при этом мешает (выключит интернет, когда наш VPN выключен).
